# Arena 프론트 개선 계획 — "기본에 충실, UX 극대화" (2026-07-11)

> **제품 본질**: 6개 알고의 **투명 독립 트랙레코드**를 공개하는 대시보드.
> "기본"이란 트랙레코드 제품이 신뢰를 얻는 표준 요소(에쿼티 커브·드로다운·벤치마크·
> 비용 명시)이고, "UX 극대화"란 그것을 5초 안에 읽히게 만드는 것이다.
> 대상: `arena/index.html` (단일 파일, lightweight-charts, Supabase 직결 — 이 구조는 유지).

## 1. 리서치 근거 (적용 원칙 5개)

| 원칙 | 내용 | 적용 |
|---|---|---|
| **5초 룰** | 대시보드는 5초 안에 핵심 메시지("누가 이기고 있나")가 읽혀야 함 | 리더보드 정렬 + 순위 표시 |
| **에쿼티 커브 = 트랙레코드의 기본** | 최종 손익 숫자만으론 경로(드로다운 깊이·기간)가 안 보임. 커브+드로다운 오버레이가 표준 | 알고별 스파크라인 + 전체 커브 |
| **벤치마크 없는 수익률은 무의미** | 같은 기간 buy&hold 대비가 없으면 -1%가 잘한 건지 못한 건지 판단 불가 | BTC 보유 대비선 |
| **점진 공개(progressive disclosure)** | 요약 먼저, 상세는 요청 시. 지표 과다는 2주 안에 대시보드를 죽임 | 카드 지표 4개 상한, 클릭 시 상세 |
| **Data-ink ratio** | 정보 없는 픽셀 제거 (이미 실천 중 — diagnostics 섹션 제거가 그 사례) | 신규 요소도 동일 기준 |

(5초 룰·역피라미드), [DataCamp 대시보드 설계](https://www.datacamp.com/tutorial/dashboard-design-tutorial)
(점진 공개), [Vizule 베스트 프랙티스](https://vizule.io/dashboard-design-best-practices/) (data-ink).

## 2. 현행 감사 (2026-07-11 기준)

**있는 것 (잘 갖춰짐)**: 헤더(BTC 현재가·LIVE·KST), 포트폴리오 바(총자산·진입중),
알고 카드 6개(자산·수익률·업타임·승률·대기사유 힌트), 캔들차트+알고 탭+진입/청산 마커,
오픈 포지션 스트립, 거래 로그 테이블, 스켈레톤 로딩, aria/반응형/reduced-motion.

**없는 것 (갭)**:

| # | 갭 | 왜 "기본"인가 |
|---|---|---|
| G1 | 에쿼티 커브·드로다운 없음 — 현재 자산 "숫자"만 | 트랙레코드 제품의 1번 요소. 경로 없이 신뢰 없음 |
| G2 | 벤치마크(buy&hold) 없음 | -0.3% 시장에서 -1%는 실패, +5% 시장에서 -1%는 더 실패 — 기준 없이 판단 불가 |
| G3 | 카드가 고정 순서 — 순위 없음 | 제품명이 **Arena**인데 누가 이기는지 정렬돼 있지 않음 (5초 룰 위반) |
| G4 | MaxDD 미표시 | 수익률만으론 리스크가 안 보임. 승률+업타임은 있는데 정작 MaxDD가 없음 |
| G5 | 거래 로그에 close_reason·비중 없음 | target_exit/trailing/time_stop 구분은 전략 투명성의 핵심 (백엔드에 이미 존재) |
| G6 | 비용·규칙 명시 없음 | 왕복 ~13bps 반영 여부, 시작일, 4h 봉 기준 — 트랙레코드 신뢰의 전제 |
| G7 | 데이터 신선도 표시가 명목뿐 | LIVE 배지는 있으나 "마지막 4h 판단 시각"이 없음 (STALE 여부 판별 불가) |
| G8 | 카드↔차트 탭 비연동 | 카드 클릭해도 아무 일 없음 — 상세로 가는 자연스러운 경로 부재 |

## 3. 개선 계획

### P0 — 기본기 (트랙레코드 신뢰 요소) ★최우선

**F-1. 에쿼티 커브 + 드로다운** (G1)
- 전체(6알고 합산) 커브를 포트폴리오 바 아래 컴팩트 영역(높이 ~160px)에 lightweight-charts
  area 시리즈로. 드로다운은 동일 차트의 하단 오버레이(음수 영역) 또는 호버 툴팁.
- 알고 카드에는 **스파크라인**(SVG 폴리라인, 라이브러리 불요, ~30줄)만 — 카드는 요약이므로.
- 데이터: 이미 fetch하는 `positions` 500건에서 청산 거래를 시간순 누적하면 프론트 계산 가능
  (`eq *= 1 + w·ret`). 추가 쿼리 불필요.

**F-2. buy&hold 벤치마크** (G2)
- 아레나 시작일(2026-06-20) 이후 BTC 수익률 라인을 F-1 전체 커브에 점선 오버레이.
- 데이터: 이미 로드하는 Binance klines 재사용. 포트폴리오 바에도 "vs BTC 보유 +X.X%p" 한 줄.

**F-3. 리더보드 정렬 + 순위** (G3)
- 알고 카드를 **총수익률 내림차순 정렬**, 좌상단에 `#1`~`#6` 순위 배지. 1위 카드 보더 강조(금색).
- 정렬 기준은 단순하게 하나(총수익률)만 — 커스텀 정렬 옵션은 지표 과다와 같은 함정.
- 진입 0건 알고는 "관망 중" 그룹으로 뒤에 (0% ≠ 꼴찌·1위 방지, 현재 시장 국면상 정당한 대기).

**F-4. 규칙·비용 투명성 라인** (G6)
- 포트바 라벨 확장: "알고당 $1,000 · 현물 long/flat · 왕복비용 ~13bps 반영 · 2026-06-20 시작(D+N)".
- 푸터의 strategy_version은 유지.

### P1 — 스캔성 (읽는 속도)

**F-5. 카드 지표 재구성** (G4) — 상한 4개 유지
- 현재: 업타임·승률 → 변경: **승률(n) · MaxDD · 거래수**+스파크라인(F-1).
  업타임은 상세(F-8)로 강등 — 일반 방문자에게 노출률은 2차 지표.
- MaxDD는 청산 거래 누적 커브에서 파생(프론트 계산, F-1과 동일 데이터).

**F-6. 거래 로그 close_reason 배지 + 비중** (G5)
- `target_exit`(초록)/`trailing_stop`(청록)/`stop_loss`(빨강)/`time_stop`(회색)/`flat_signal`(무채색)
  칩. 비중 컬럼(w=0.46 등) 추가.
- ⚠️ 선행 작업: `arena_spot_position_mart_v1` 뷰에 `close_reason` 컬럼 노출 필요(뷰 수정 1회).

**F-7. 신선도 명시** (G7)
- 헤더 LIVE 배지 옆 또는 푸터에 "마지막 판단: N분 전 (4h 주기)". `arena_decisions`
  최신 created_at은 이미 fetch 중(fetchDecisions) — 표시만 추가.
- 4h+30m 초과 시 배지를 STALE(주황)로 전환 — 정직한 상태 표시가 LIVE 배지의 신뢰를 만듦.

### P2 — 점진 공개 (상세 경로)

**F-8. 카드 → 차트/로그 연동** (G8)
- 카드 클릭 = 해당 알고 차트 탭 활성화 + 거래 로그 필터. 이미 탭 메커니즘이 있으므로
  이벤트 연결만. 모바일에서 특히 유효(카드가 첫 화면, 차트는 스크롤 아래).

**F-9. 기간 토글 (ALL / 30D / 7D)**
- 카드 수익률·커브·로그에 공통 적용되는 단일 토글. 기본 ALL.
  트랙레코드는 전기간이 기본이므로 ALL 우선 — 최근 개선(파라미터 버전업) 확인용 보조.

### 하지 않을 것 (기본 유지 원칙)

- ❌ SPA/프레임워크 전환, 빌드 체인 도입 — 단일 HTML + CDN 2개 구조가 이 규모의 정답.
- ❌ 지표 추가 요구가 와도 카드 4개 상한 초과 금지 (expectancy·PF·Sharpe는 F-8 상세로).
- ❌ decision diagnostics 유형의 raw 진단 재도입 (2026-07-11 제거 — 방문자용 아님).
- ❌ 실시간 틱 스트리밍 확대 — 4H 전략에 초단위 갱신은 과잉 (현재 60s 폴링 유지).

## 4. 구현 노트

- **추가 쿼리 0개로 P0 전부 가능**: 커브·MaxDD·벤치마크·정렬 모두 기존 fetch
  (positions 500건 + klines) 파생. F-6만 mart 뷰 수정 1회 필요.
- 스파크라인은 인라인 SVG(외부 라이브러리 금지), 전체 커브만 기존 lightweight-charts 재사용.
- 배포: `npx wrangler pages deploy arena/ --project-name arena --branch=main` (Production은
  `--branch=main` 필수 — 기본값은 Preview로 감).
- 예상 공수: P0 = 1세션, P1 = 0.5세션(+뷰 수정), P2 = 0.5세션.

## 5. 실행 순서

```
1차   F-3 리더보드 정렬 (최소 변경·최대 효과) → F-4 투명성 라인 → F-1 커브 → F-2 벤치마크
2차   F-5 카드 지표 → F-7 신선도 → F-6 close_reason(뷰 수정 포함)
3차   F-8 카드 연동 → F-9 기간 토글
```
