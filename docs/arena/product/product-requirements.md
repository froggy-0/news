# BTC Signal Arena — 제품 요구사항 문서 (PRD)

> 최초 작성: 2026-06-19
> 성격: 기능 요구사항 + 사용자 경험 기준 (비기술)

---

## 1. 제품 정의

BTC Signal Arena는 세 가지 레이어로 구성된다:

```
레이어 1: 데이터 수집    — 거시 + 기술 지표 실시간 수집
레이어 2: 알고리즘 엔진  — 5개 전략이 독립적으로 신호 생성 + 가상 거래
레이어 3: 공개 인터페이스 — 트랙레코드 리더보드 + 신호 알림
```

---

## 2. 핵심 요구사항 (Must Have)

### 2-1. 불변 트랙레코드

**요구사항:** 한 번 기록된 포지션은 수정·삭제 불가

- 신호 발생 즉시 Supabase에 `open_time`, `open_price`, `algo_id`, `direction` 기록
- 청산 시 `close_time`, `close_price`, `ret_pct`, `hit` 추가 기록
- DB 레벨에서 UPDATE 제한 (closed 레코드의 핵심 필드)
- 관리자도 과거 기록을 소급 수정할 수 없는 구조

**이유:** 트랙레코드의 신뢰는 "손대지 않았다"는 증명에서 나온다.

### 2-2. 실시간 거래 반영

**요구사항:** 페이퍼 트레이딩이 실제 시장과 동기화되어야 함

- 진입 가격: 신호 발생 시점 Binance 4H 봉 종가 (실제 거래 가능한 가격)
- 청산 가격: 스톱로스 = 실시간 Binance 1m 봉 가격 (지연 없음)
- 청산 가격: 신호 반전 = 4H 사이클 시점 가격
- 수수료 반영: 5bps/leg (Binance taker fee 기준)

**이유:** "어떤 알고리즘이 더 좋은가"를 평가하려면 동일한 시장 조건이어야 한다.

### 2-3. 알고리즘 투명성

**요구사항:** 각 알고리즘의 신호 조건이 문서로 공개되어야 함

- 예: "regime_v2: BullQuiet 상태일 때만 long 포지션 진입"
- 예: "multi_factor: BullQuiet AND RSI<50 AND MACD양수 동시 충족 시 long, BearPanic AND RSI>55 AND MACD음수 동시 충족 시 short"
- 사용자가 "왜 이 신호가 나왔는가"를 이해할 수 있어야 함

**이유:** 블랙박스 AI 신호에 대한 불신이 높다. 논리 공개가 신뢰의 근거.

### 2-4. 멀티 알고리즘 동시 운영

**요구사항:** 5개 알고리즘이 완전히 독립적으로 작동

- 각 알고리즘은 자신의 포지션만 관리 (서로 영향 없음)
- 동일한 시장 데이터를 받아 각자 다른 결론을 낼 수 있음
- 알고리즘 간 성과 비교가 공정하게 이루어져야 함

### 2-5. ATR 기반 스톱로스 (손실 제한)

**요구사항:** 포지션별 진입 시점 ATR(14)에 기반해 절대 손절가를 저장하고, 가격이 도달하면 즉시 자동 청산

- Binance WebSocket 1분 봉 실시간 감지
- 장 중 급락/급등 시에도 지연 없이 체결 (4H 사이클 대기 없음)
- 손절 거리: `ATR(14) * ATR_MULTIPLE`, 기본 2.5배
- 손절 비율 상하한: 기본 2%~8% 클램프
- 서버 재시작 후에도 최초 진입 시 저장한 `stop_loss_price`를 우선 사용
- 스톱로스로 청산된 포지션은 `is_stop_loss=true`로 별도 구분

## 3. 중요 요구사항 (Should Have)

### 3-1. 알고리즘 신호 방향

- 각 알고리즘이 `long` / `short` / `flat` 세 가지 상태를 명확히 구분
- `flat`은 포지션 없음 (시장 관망)
- `flat`에서 무리하게 신호를 강제하지 않음 (0 포지션도 전략)

### 3-2. 정확한 수수료 계산

- 진입 수수료 + 청산 수수료 각각 5bps = 10bps 총 비용 반영
- 수익률 계산: `sign * (exit_price / open_price - 1) - 0.10%`
- 단순 가격 변동이 아닌 실제 거래 비용 반영

### 3-3. 포지션 상태 가시성

내부 모니터링을 위한 현재 포지션 현황:

```
현재 오픈 포지션:
  regime_v2:       long  @ $105,200  (진입 2026-06-19 04:05)
  fng_contrarian:  flat  (포지션 없음)
  vix_rsi:         long  @ $104,800  (진입 2026-06-18 20:05)
  macd_momentum:   short @ $106,100  (진입 2026-06-19 00:05)
  multi_factor:    flat  (포지션 없음)
```

### 3-4. 장애 복구

- 서버 재시작 시 DB에서 오픈 포지션 자동 로드 (상태 복구)
- WebSocket 연결 끊어져도 자동 재접속 (5초 간격)
- 4H 사이클 실패해도 다음 사이클에서 정상 재개

- 신호 변경 시 텔레그램 봇 메시지 발송

- 손절 발생 시: "⚠️ regime_v2: 스톱로스 청산 @ $99,500 (손실 -5.1%)"

## 5. 절대 하지 않을 것 (Anti-requirements)

| 항목 | 이유 |
|---|---|
| 과거 기록 수정 | 신뢰 자산 파괴 |
| 알고리즘 조건 숨김 | 차별화 포인트 훼손 |
| 승률 높은 거래만 선택 공개 | 근본 가치관 위반 |
| 실제 고객 자금 운용 | Phase 4 전 법적 검토 없이 불가 |
| "이 신호대로 거래하세요" 표현 | 무등록 투자자문 리스크 |
| 24시간 인간 개입 | 알고리즘 자동화 원칙 위반 |

## 6. 사용자 경험 기준

### 신뢰를 주는 표현 방식

**올바른 표현:**
- "이 알고리즘은 지난 6개월간 342회 신호를 냈으며, 이 중 62%에서 수익이 발생했습니다."
- "2026년 5~6월 하락장에서 macd_momentum은 총 -12.4%의 손실을 기록했습니다."

**피해야 할 표현:**
- "이 신호를 따라 매매하면 수익을 올릴 수 있습니다."
- "전문가 추천 매수 신호"
- "적중률 90%"

### 손실 데이터 처리 원칙

- 손실 거래를 작게 표시하거나 흐리게 하지 않는다
- 손실이 심한 알고리즘도 동일한 방식으로 표시
- "이 알고리즘은 이런 시장 환경에서 실패합니다"를 명확히 표시

## 8. 알고리즘 교체 정책

포워드 테스트 중 알고리즘을 수정할 경우:

1. **기존 버전 포지션 유지**: 수정 전 오픈 포지션은 수정 전 버전 로직으로 청산
2. **새 버전은 신규 레코드**: `algo_id`에 버전 접미사 추가 (`regime_v2_v3`)
3. **비교 가능성 유지**: 두 버전이 동시에 기록되어 A/B 비교 가능
4. **소급 수정 금지**: 과거 포지션의 진입/청산 가격 수정 불가
