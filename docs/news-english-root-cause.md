# 핵심 뉴스 영어 잔존 원인 분석

## 현상

`email.log` 기준, "■ 핵심 뉴스" 섹션의 5개 항목 모두:
- 헤드라인이 영어 (예: `SEC Issues Major Interpretation Clarifying Federal Securities Laws for Crypto Assets`)
- 본문(서술 단락)이 영어
- "핵심 한줄" 요약이 영어 헤드라인을 그대로 반복

프롬프트에서 `① 한국어 헤드라인 — 출처명`을 명시적으로 요구하고 있음에도 LLM이 따르지 않고 있다.

---

## 데이터 흐름 추적

```
뉴스 수집 (Perplexity/Grok/NewsAPI)
  → 영어 title, summary, why_it_matters
    → packet["news"] 에 저장
      → news_focus_json 으로 직렬화
        → brief_input.j2 에서 LLM에 전달
          → LLM이 Section 4-2 생성 (여기서 한국어로 변환해야 함)
            → validate_and_rewrite_briefing() 검수/재작성
              → parse_news_items() 파싱
                → email_news.html.j2 렌더링
```

## 원인 분석

### 1차 원인: LLM 입력 데이터가 전부 영어

`news_focus_json`의 `top_items`에 들어가는 필드:
- `title`: 영어 원문 그대로 (예: "SEC Issues Major Interpretation...")
- `summary`: 영어 원문
- `why_it_matters`: 영어 원문
- `citations`: 영어 URL

뉴스 수집 단계(`build_news_packet`)에서 Perplexity, Grok, NewsAPI 모두 영어 소스를 수집하므로, LLM에 전달되는 뉴스 데이터 자체가 100% 영어다.

### 2차 원인: 프롬프트 지시 강도 부족

`brief_instructions.j2`에서 한국어 헤드라인을 요구하는 부분:

```
4-2. 핵심 뉴스 5선
   - ①~⑤ 번호를 붙여 최대 5개 뉴스를 다뤄요
   - 각 뉴스 형식:
     ① 한국어 헤드라인 — 출처명
```

`brief_input.j2`에서도:

```
- Section 4-2 핵심 뉴스 5선: ①~⑤ 번호 + 헤드라인 — 출처명 + 서술 5~8문장 + 원문 링크 + 핵심 한줄 형식이에요.
```

문제점:
- "한국어 헤드라인"이라고만 적혀있고, **영어 원문을 한국어로 번역하라**는 명시적 지시가 없음
- 본문(서술 단락)과 "핵심 한줄"에 대해서는 한국어 작성 지시가 아예 없음
- LLM은 입력 데이터(영어)를 그대로 옮기는 것이 가장 "안전한" 선택이므로, 강한 지시 없이는 영어를 유지하는 경향이 있음

### 3차 원인: 검수(validator)가 영어를 탐지하지 못함

`brief_validator_instructions.j2`의 관련 규칙:

```
- Section 4-2 각 항목에는 한국어 헤드라인, 서술 단락, 원문 링크, 핵심 한줄이 있어야 한다.
```

문제점:
- "한국어 헤드라인이 있어야 한다"고만 되어있고, **영어 헤드라인이면 실패**라는 명시적 규칙이 없음
- 영어 헤드라인이 존재하면 "헤드라인이 있다"는 조건은 충족하므로 pass 처리됨
- 본문이 영어인지 검사하는 규칙 자체가 없음

### 4차 원인: 재작성(rewrite)도 영어를 한국어로 바꾸지 않음

`brief_rewrite_instructions.j2`:

```
- Section 4-2 각 항목에는 한국어 헤드라인, 서술 단락, 원문 링크, 핵심 한줄을 유지한다.
```

"유지한다"는 표현이 문제 — 이미 영어로 작성된 것을 "유지"하게 됨.

### 5차 원인: 파싱/렌더링 단계에 번역 로직 없음

`parse_news_items()`는 LLM 출력을 그대로 파싱만 하고, `email_news.html.j2`도 받은 데이터를 그대로 렌더링한다. 어느 단계에서도 영어→한국어 변환이 일어나지 않는다.

---

## 영향 범위

| 필드 | 현재 상태 | 기대 상태 |
|------|----------|----------|
| 헤드라인 (①~⑤) | 영어 원문 | 한국어 번역 |
| 서술 단락 (body) | 영어 원문 | 한국어 해석/설명 |
| 핵심 한줄 (tldr) | 영어 헤드라인 반복 | 한국어 한 문장 요약 |
| 출처명 | 영어 (정상) | 영어 유지 (원래 의도) |
| 원문 링크 | URL (정상) | URL 유지 (원래 의도) |

---

## 개선 방안

### A. 프롬프트 강화 (권장, 가장 효과적)

`brief_instructions.j2`의 Section 4-2 지시를 강화:

```
4-2. 핵심 뉴스 5선
   - 입력 데이터의 뉴스 제목과 본문은 영어 원문이에요.
   - 반드시 한국어로 번역/의역하여 작성해요. 영어 원문을 그대로 쓰면 안 돼요.
   - 각 뉴스 형식:
     ① 한국어 헤드라인 (영어 원문 번역) — 출처명
     한국어 서술 단락 (5~8문장)
     → 원문 링크 URL
     핵심 한줄: 한국어 한 문장 요약
   - 헤드라인, 서술 단락, 핵심 한줄 모두 반드시 한국어로 작성해요.
   - 출처명과 원문 링크만 영어를 허용해요.
```

`brief_input.j2`에도 동일 강화:

```
- Section 4-2: 뉴스 데이터는 영어 원문이지만, 출력은 반드시 한국어로 번역하여 작성해요.
  헤드라인, 서술 단락, 핵심 한줄 모두 한국어여야 해요. 영어 원문 그대로 쓰면 안 돼요.
```

### B. 검수 규칙 강화

`brief_validator_instructions.j2`에 영어 탐지 규칙 추가:

```
- Section 4-2의 헤드라인, 서술 단락, 핵심 한줄이 영어로 작성되어 있으면 실패다.
  출처명과 원문 링크만 영어를 허용한다.
```

### C. 재작성 지시 수정

`brief_rewrite_instructions.j2`에서 "유지한다" → "한국어로 작성한다"로 변경:

```
- Section 4-2 각 항목의 헤드라인, 서술 단락, 핵심 한줄은 반드시 한국어로 작성한다.
  영어로 되어 있으면 한국어로 번역하여 다시 쓴다.
```

### D. 파싱 단계 방어 (선택)

`parse_news_items()`에서 영어 비율이 높은 항목에 경고 로그를 남기는 방어 로직 추가 가능.
단, 이 단계에서 번역하는 것은 LLM 호출이 추가로 필요하므로 비용/지연 측면에서 비권장.
