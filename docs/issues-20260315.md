## P0 — 뉴스 0건으로 브리핑 발송

### 현상

Perplexity Search API 15건 요청 전부 HTTP 200 성공했지만, 반환된 결과가 **전부 후처리 필터에서 탈락**하여 최종 뉴스 0건. legacy fallback(RSS+NewsAPI)도 0건, OpenAI web_search backfill도 0건. `data_quality: "critical"`인데 `status: "ok"`로 브리핑이 발송됨.

### 로그 증거

```
macro     → raw 3건 → jp.reuters.com(1), sponsored.bloomberg.com(2) → 전부 차단 → 0건
us_equity → raw 2건 → partners.wsj.com(1), jp.reuters.com(1)       → 전부 차단 → 0건
ai_bigtech→ raw 4건 → jp.reuters.com(4) 일본 종목 quote 페이지     → 전부 차단 → 0건
bitcoin   → raw 2건 → data.coindesk.com(2) 데이터 상품 페이지      → 전부 차단 → 0건
```

### 원인 분석

**1) Perplexity가 비영어/비기사 결과를 반환하는 구조적 문제**

`search_domain_filter`에 `reuters.com`을 allowlist로 넣으면 서브도메인 `jp.reuters.com`도 API 레벨에서 통과합니다. 공식 문서: "Specify a root domain to match all content from that domain and its subdomains." 마찬가지로 `bloomberg.com` → `sponsored.bloomberg.com`, `wsj.com` → `partners.wsj.com`도 통과.

코드의 후처리 필터(`EXCLUDE_URL_PATTERNS`, `DISALLOWED_SPONSORED_DOMAINS` 등)가 이들을 정확히 차단하고 있지만, **API가 이런 결과만 반환하면 전멸**합니다.

```python
# perplexity_search.py — 현재 domain_filter
domain_filter=(
    "reuters.com",           # → jp.reuters.com도 매칭
    "bloomberg.com",         # → sponsored.bloomberg.com도 매칭
    "wsj.com",               # → partners.wsj.com, cn.wsj.com도 매칭
    ...
)
```

**2) 주말 실행 — 영어 금융 뉴스 부재**

2026-03-15(일요일) UTC 07:34 실행. 미국 시장 휴장으로 최신 영어 금융 기사가 극히 적어 Perplexity가 관련성 낮은 결과로 채움.

**3) retry 전략이 시간만 넓히고 도메인 전략 불변**

```
1차: recency="day"                    → 0건
2차: last_updated 2일 범위            → 0건 (같은 저품질 결과)
3차: search_after/before 2일 범위     → 0건 (같은 저품질 결과)
4차: recency="week" + 넓은 도메인     → 0건 (여전히 jp.reuters.com만)
```

4단계 retry 모두 동일한 allowlist를 사용하므로 Perplexity가 같은 저품질 결과를 반복 반환.

**4) `news_count == 0`인데 `status: "ok"`**

```python
# pipeline.py — status 결정 로직
# BriefGenerationError → "openai_failed"
# 일반 Exception → "failed"
# brief_fallback_used → "brief_fallback"
# 그 외 → "ok"  ← 뉴스 0건이어도 여기에 해당
```

`data_quality`가 `"critical"`이어도 파이프라인 status에 반영되지 않음.

### 영향

- 뉴스 없이 시장 데이터만으로 브리핑 생성 → 내용 품질 저하
- 수신자가 "뉴스 0건 브리핑"을 정상 브리핑으로 오인
- 검수(review)에서도 "데이터 부족 경고와 뉴스 항목 충돌" 지적했지만 발송 차단 안 됨

### 제안

1. `data_quality == "critical"`이면 `status`를 `"degraded"` 이상으로 설정하고, 이메일 제목에 `[데이터 부족]` 표시 또는 발송 스킵
2. `search_domain_filter`에 `-jp.reuters.com`, `-sponsored.bloomberg.com`, `-partners.wsj.com`, `-cn.wsj.com` denylist 추가 (allowlist와 동시 사용 불가이므로, 후처리 강화 또는 retry 시 denylist 모드 전환)
3. 주말/휴일 감지 시 `recency_filter`를 `"week"`으로 시작
4. broad retry 단계에서 query 자체를 변경하는 전략 추가 (현재는 `retry_query`가 있지만 결과 도메인 분포가 동일)

## P1 — Perplexity domain_filter의 이중 역할 혼란

### 현상

`SearchTopic.domain_filter`가 두 가지 역할을 겸합니다:
- Perplexity Search API의 `search_domain_filter` 파라미터로 전달 (API 레벨 필터)
- `_parse_results()`에서 `_is_allowed_domain()` 결과 후처리 필터

그런데 URL prefix 형태(`https://www.ft.com/content/`, `https://investor.nvidia.com` 등)는 path가 포함되어 있어 `_search_domain_filter_values()`에서 **API에는 전달되지 않고** 후처리에서만 작동합니다.

### 코드 증거

```python
# _search_domain_filter_values() — path가 있으면 skip
parsed = urlparse(body if "://" in body else f"https://{body}")
if parsed.path not in {"", "/"}:
    continue  # "https://www.ft.com/content/" → 탈락

# 하지만 _is_allowed_domain()에서는 URL prefix 매칭으로 사용
def _is_allowed_domain(url: str, allowed_domains: tuple[str, ...]) -> bool:
    if not any(_matches_source_filter(url, candidate) for candidate in allowed_domains):
        return False  # URL prefix도 여기서 매칭
```

### 영향

- `FT_CONTENT_URL_PREFIX`(`https://www.ft.com/content/`)가 API 필터에 빠져 있어, FT 기사가 API 결과에 포함되려면 다른 도메인 필터(`ft.com` 등)에 의존해야 함 → 실제로 `ft.com`이 없는 토픽에서는 FT 기사가 아예 안 옴
- `retry_domain_filter`의 회사 IR 도메인들(`investor.nvidia.com`, `news.microsoft.com/source/` 등)도 API에 전달 안 됨 → retry 시 도메인 범위가 넓어지는 효과 없음

### 제안

- API 전달용 도메인 목록과 결과 후처리용 URL prefix를 별도 필드로 분리
- 또는 `domain_filter`에서 path 포함 항목은 도메인 부분만 추출해서 API에도 전달

## P2 — BTC ETF Perplexity 참조가 비공식 API 경로 사용

### 현상

`btc_etf_official.py`가 Sonar Chat Completions API에 Search API 전용 파라미터를 전달:

```python
client.chat.completions.create(
    model="sonar",
    search_domain_filter=list(BTC_ETF_REFERENCE_DOMAINS),  # Search API 파라미터
    search_recency_filter="month",                          # Search API 파라미터
    search_mode="web",                                      # 문서에 없음
    response_format={"type": "json_schema", ...},           # Agent API 권장
)
```

### 로그 증거

이번 실행에서 빈 응답 반환:
```json
{"event": "btc_etf_reference_parse_empty", "response_preview": "{\"snapshots\": []}"}
```

direct fetch fallback으로 IBIT/BITB는 성공했지만 GBTC는 429 rate limit.

### 영향

- 비공식 파라미터 사용으로 API 변경 시 예고 없이 깨질 수 있음
- 주말에 issuer 페이지 업데이트가 없으면 Sonar가 빈 배열 반환 → direct fetch에 의존
- Grayscale(`etfs.grayscale.com`) 429 rate limit이 반복되면 GBTC 데이터 누락

### 제안

- Perplexity Agent API로 마이그레이션 (structured output 공식 경로)
- 또는 Sonar 경로를 제거하고 direct fetch를 primary로 승격

## P3 — OpenAI web_search backfill도 전멸

### 현상

Perplexity 실패 후 OpenAI web_search backfill이 실행됐지만 기사 0건:

```json
{
  "event": "web_backfill_result",
  "citation_count": 8,
  "extra_item_count": 0,
  "reason": "no_items_parsed",
  "citation_samples": [
    "https://bloomberg.com/authors/AOWVrRAsa3I/toshiro-hasegawa",
    "https://bloomberg.com/latest/ecb",
    "https://bloomberg.com/authors/AOW9mDgCGr8/lucy-meakin",
    ...
  ]
}
```

### 원인

OpenAI web_search가 반환한 citation 8건이 전부 Bloomberg 저자 프로필 페이지(`/authors/...`)와 토픽 랜딩 페이지(`/latest/...`). `BACKFILL_SOURCE_EXCLUDE_PATTERNS`에 `/authors/`가 포함되어 있어 전부 차단.

### 영향

3중 fallback(Perplexity → legacy RSS/NewsAPI → OpenAI web_search) 전부 실패. 주말에는 모든 경로가 동시에 실패할 수 있는 구조적 취약점.

### 제안

- 주말/휴일에는 backfill query를 "주간 요약" 스타일로 변경
- 또는 뉴스 0건 시 브리핑을 "시장 데이터 전용 모드"로 전환하고 명시

## P4 — `_warn_once` 전역 상태 미초기화

### 현상

`market.py`의 `_provider_warned: set[str]`가 모듈 레벨 전역 변수. `reset_provider_runtime_state()`가 `provider_runtime.py`의 상태만 초기화하고 이 set은 리셋하지 않음.

### 영향

- `main.py schedule` 모드(매일 반복)에서 두 번째 실행부터 provider 경고가 누락
- GitHub Actions는 매번 새 프로세스이므로 영향 없음 (이번 로그에서는 해당 없음)
- 로컬 스케줄러 운영 시에만 문제

### 제안

- `reset_provider_runtime_state()`에서 `market._provider_warned.clear()` 호출 추가
- 또는 `_provider_warned`를 `provider_runtime.py`로 이동

## P5 — BTC ETF stale 캐시 무한 재사용

### 현상

이번 실행에서 `btc-etf-snapshots-20260315` 캐시 hit. Perplexity가 빈 응답을 반환했을 때 이전 캐시를 그대로 사용:

```python
# market.py
if not snapshots:
```

### 영향

- Perplexity가 며칠간 실패하면 오래된 캐시가 "현재 보유량"으로 브리핑에 표시

- observability 로그에 캐시 나이 기록

## P6 — 검수(review) 지적이 발송을 차단하지 않음

### 현상

브리핑 검수에서 3건의 보완점을 찾고, 재작성 1회 후에도 보완점이 남았지만 그대로 발송:

```
검수 1차: "구조 마지막 부분이 잘려 문장이 미완성", "출처 표기 불일치", "해석 충돌"
재작성 후 검수: "데이터부족 문구와 항목 충돌로 모순", "단정적 인과 표현", "출처 표기 위치 불일치"
→ 그대로 발송
```

### 원인

`_run_rewrite_loop()`에서 `max_rewrites`(1회) 소진 후 현재 버전을 반환. 검수 실패가 발송 차단 조건이 아님.

### 영향

- 품질 미달 브리핑이 수신자에게 전달
- 특히 "데이터 부족 경고와 뉴스 항목 충돌로 모순"이라는 검수 지적이 있는 상태로 발송

### 제안

- 검수에서 `structure_pass == false`이면 발송 차단 또는 fallback 브리핑으로 대체
- 또는 `data_quality == "critical"` + 검수 실패 조합 시 발송 스킵
