# Development Standards

이 문서는 이 저장소의 공통 개발 기준입니다. `AGENTS.md`는 이 문서를 요약한 실행 규칙이고, 실제 품질 강제는 `pyproject.toml`, `pre-commit`, GitHub Actions로 처리합니다.

## 1. 운영 원칙

- 작은 변경으로 문제를 해결하고, 무관한 리팩터는 섞지 않습니다.
- Python 코드는 CI 기준인 Python 3.11 호환성을 유지합니다.
- 외부 데이터 공급자 변경은 API 계약 변경으로 취급합니다.
- 경고 로그는 단순 잡음이 아니라 운영 신호여야 합니다.
- 같은 판단을 여러 곳에서 복제하지 말고 정책 계층으로 모읍니다.

## 2. AI 에이전트 기준

이 저장소는 OpenAI Codex의 `AGENTS.md` 계층 규칙을 기준으로 구성합니다.

- 루트 `AGENTS.md`는 짧게 유지하고, 상세 기준은 이 문서에 둡니다.
- 디렉터리별 특수 규칙이 필요할 때만 하위 `AGENTS.md`를 추가합니다.
- 권한 제어는 도구 설정과 CI에 맡기고, 문서에는 행동 기준만 적습니다.
- 자동 검증에 통과하지 못하는 변경은 완료로 보지 않습니다.

## 3. 작업 흐름

1. 관련 코드, 테스트, 문서를 읽고 변경 범위를 정합니다.
2. 구현은 하나의 책임 단위로 나눠 진행합니다.
3. 동작이 달라지면 테스트를 먼저 추가하거나 함께 수정합니다.
4. `make fmt`, `make lint`, `make test`, `make typecheck`, `make check` 순서로 검증합니다.
5. 설정, 환경변수, 운영 절차가 바뀌면 `README.md`를 갱신합니다.

## 4. Python 코드 규칙

- 공개 함수와 경계 타입은 가능한 한 명시적 타입 힌트를 유지합니다.
- 복잡한 분기나 파싱 로직은 작은 함수로 분리하고 테스트로 고정합니다.
- `print` 대신 `logging`을 사용하고, 운영 판단에 필요한 문맥을 함께 남깁니다.
- 반복되는 문자열 규칙, 공급자 정책, 품질 기준은 상수나 전용 모듈로 모읍니다.
- 한 함수가 여러 외부 시스템에 동시에 의존한다면 책임 분리를 우선 검토합니다.

## 5. 데이터 수집 규칙

- 공급자별 요청 간격, 재시도, quota 처리, circuit breaker는 공용 정책 계층에서 관리합니다.
- HTTP와 SDK 호출은 같은 provider retry 계층을 공유하고, 지수 백오프와 `Retry-After` 처리를 한곳에서 관리합니다.
- `404` 같은 영구 실패는 재시도하지 않고, `429/5xx/timeout` 중심으로만 재시도합니다.
- fallback은 “가능하면 뭔가 가져온다”보다 “같은 실행에서 일관된 기준으로 가져온다”를 우선합니다.
- 집계 로직은 부분 성공을 허용해 한 소스 실패가 전체 결과를 무너뜨리지 않도록 설계합니다.
- fallback, ranking, citation 정책을 바꿀 때는 provider 로그와 품질 평가 테스트를 함께 갱신합니다.

## 6. 테스트 기준

- 동작을 바꾸는 변경은 최소 한 개 이상의 회귀 테스트를 포함합니다.
- 파서 변경은 fixture 또는 실제 응답에 가까운 샘플로 고정합니다.
- fallback 정책 변경은 “어떤 경고가 사라지고 어떤 경고가 남아야 하는지”까지 테스트합니다.
- formatter/linter는 기본 품질선이고, 비즈니스 규칙 검증은 pytest로 보완합니다.
- 변경 유형별 집중 검증 기준은 `docs/ai-evals.md`를 따릅니다.

## 7. 문서와 설정

- 신규 환경변수, 기본값 변경, 워크플로우 변경은 `README.md`에 반영합니다.
- AI 에이전트 규칙을 바꾸면 `AGENTS.md`와 이 문서가 서로 모순되지 않아야 합니다.
- 개인 전용 설정은 커밋하지 않고, 저장소에는 팀 공용 기준만 둡니다.
- 기능별 요구사항, 설계, 작업 체크리스트는 `docs/specs/` 아래에 둡니다.
- 보고서, 리서치 노트, 실행 분석 문서는 각각 `docs/reports/`, `docs/research/`, `docs/analysis/` 아래에 둡니다.

## 8. 커밋 규칙

- 제목 형식: `type(scope): 한국어 요약`
- 권장 `type`: `feat`, `fix`, `refactor`, `perf`, `test`, `docs`, `ci`, `chore`
- 제목은 현재형으로 짧게 씁니다.
- 본문에는 아래 세 항목을 남깁니다.

```text
배경:
- 왜 이 변경이 필요한가

변경:
- 핵심 변경점

검증:
- 실행한 명령과 결과
```

## 9. 리뷰 루브릭

변경을 마무리하기 전에 아래 질문을 확인합니다.

- 정확성: 예전보다 더 정확한 결과를 내는가
- 일관성: fallback과 로그가 실행마다 흔들리지 않는가
- 격리성: 한 공급자 실패가 전체 결과를 과도하게 오염시키지 않는가
- 가시성: 로그와 테스트만으로 실패 원인을 좁힐 수 있는가
- 유지보수성: 중복 규칙을 줄이고 책임 경계를 더 분명히 했는가
- 문서성: 운영 문서와 개발 문서가 실제 동작과 맞는가
