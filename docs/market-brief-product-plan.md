# Morning Market Brief 데이터 제품 기획안

## 1. 문서 목적
이 문서는 Morning Market Brief를 단순 뉴스 요약 메일이 아니라, 매일 아침 빠르게 읽고 바로 판단할 수 있는 `시장 정보 제품`으로 고도화하기 위한 기획안이다.

핵심 목표는 세 가지다.
- 신뢰 가능한 데이터를 역할별로 분리해 중복 없이 수집한다.
- 수집한 데이터를 `판단 가능한 정보`로 분석하고 구조화한다.
- 이메일과 향후 웹 화면에서 같은 정보를 더 읽기 쉽고 더 신뢰감 있게 전달한다.

이 문서는 2026년 기준에 맞는 금융 브리핑 스타일, 모바일 우선 이메일 설계, 금융 데이터 시각화 베스트 프랙티스를 반영해 작성했다.

## 2. 제품 정의
### 제품 한 줄 정의
Morning Market Brief는 미국 기술주와 비트코인을 중심으로, 밤사이 시장 변화를 `숫자`, `공식 시그널`, `뉴스 해석`으로 나눠 정리해 전달하는 아침 브리핑 제품이다.

### 핵심 사용자
- 아침에 3~5분 안에 시장 분위기를 파악하고 싶은 일반 직장인
- 기술주, 반도체, 비트코인 흐름을 함께 보는 투자자
- 숫자 자체보다 `그래서 지금 시장이 어떤 분위기인지`를 빠르게 이해하고 싶은 사용자

### 사용자 가치
- 숫자만 보여주는 것이 아니라 `무슨 일이 있었는지`, `왜 중요한지`, `오늘 무엇을 봐야 하는지`까지 전달한다.
- 과도한 뉴스 나열 대신 `정리된 판단`을 제공한다.
- 출처와 구조가 분명해서 신뢰감을 준다.

## 3. 리서치 기반 핵심 인사이트
### 이메일/뉴스레터 베스트 프랙티스
- 이메일은 첫 화면에서 핵심을 파악할 수 있어야 한다.
- 모바일 우선 단일 컬럼이 가장 안정적이다.
- 긴 문단보다 `요약 -> 본문 -> 주목 포인트` 구조가 더 잘 읽힌다.
- 다크 모드 최적화보다 `깨지지 않는 밝은 기본형 + 높은 대비`가 더 실무적이다.
- 접근성을 위해 짧은 문단, 분명한 섹션 제목, plain text fallback이 필요하다.

### 금융 브리핑 톤 베스트 프랙티스
- 금융기관 브리핑은 보통 `날짜 중심 헤더`, `짧은 핵심 요약`, `시장 배경`, `주목할 변수`, `면책 문구` 흐름을 사용한다.
- 문장은 차분하고 단정해야 하지만, 지나치게 기관 보고서처럼 딱딱하면 일반 사용자는 읽기 어렵다.
- 따라서 `정제된 존댓말 + 쉬운 해석`이 최적이다.

### 데이터 시각화 베스트 프랙티스
- 금융 데이터는 `장식보다 해석`이 중요하다.
- 작은 화면에서는 차트 한 개에 모든 것을 넣기보다 `핵심 수치 + 짧은 인라인 차트 + 주석` 구조가 더 좋다.
- 사용자가 읽는 순서는 `숫자 -> 변화 -> 해석 -> 리스크`로 설계해야 한다.
- 작은 다중 차트(small multiples), 짧은 설명, 핵심 주석(annotation) 중심 구성이 2026에도 유효하다.

## 4. 제품 아키텍처: 3개 수집 루트 + 1개 편집 레이어
이 제품은 역할이 다른 세 개의 데이터 수집 루트와, 하나의 편집/검증 레이어로 구성하는 것이 가장 합리적이다.

### 루트 1. 정형 수치 레이어
역할: 숫자 원본만 담당한다.

포함 데이터:
- 미국 금리
- 미국 국채 수익률
- 달러
- VIX
- S&P500 / NASDAQ / 반도체 프록시
- 주요 기술주 종가/등락률
- 비트코인 가격
- BTC ETF 보유량 / 순유입 / 거래량
- 공포탐욕지수

대표 소스:
- FRED
- Alpha Vantage
- ETF 공식 페이지
- Alternative.me Fear & Greed

원칙:
- 숫자는 이 레이어만 소유한다.
- Perplexity나 X는 숫자 원본이 될 수 없다.
- 숫자 충돌 시 이 레이어가 최종 기준이다.

### 루트 2. 금융 리서치 레이어
역할: 시장 뉴스, 기사 맥락, 해석용 컨텍스트를 모은다.

권장 도구:
- Perplexity Search API
- 필요 시 Perplexity Sonar는 요약/리서치 메모 용도로 제한 사용

중요 원칙:
- Perplexity는 `금융 전용 수치 API`가 아니라 `리서치/검색 레이어`로 본다.
- Search는 기사 후보와 출처를 구조화해서 수집하는 용도다.
- Sonar는 요약 답변용으로 활용할 수 있지만, 숫자 원본과 공식 발표 확인을 대신하지 않는다.

이 레이어가 담당할 것:
- 주요 뉴스 후보 수집
- 뉴스 요약
- why it matters 정리
- 섹터/주제별 맥락 연결

이 레이어가 담당하지 않을 것:
- 금리/지수/BTC 가격 같은 숫자 원본
- 공식 발표 여부의 최종 확정

### 루트 3. 공식 시그널 레이어
역할: 기업, 규제기관, ETF 발행사 등의 `직접 발신 정보`를 수집한다.

포함 채널:
- 기업 newsroom / IR
- 규제기관 페이지
- ETF 발행사 공식 페이지
- 공식 X 계정

권장 운영 방식:
- `티커 전체 검색`이 아니라 `대표 엔티티 + 공식 계정 allowlist` 구조로 운영한다.
- X는 메인 뉴스 소스가 아니라 공식 입장 확인용이다.

이 레이어가 담당할 것:
- 공식 발표
- 해명/정정
- 이벤트 일정
- 규제기관 공지
- 발행사 공식 업데이트

이 레이어가 담당하지 않을 것:
- 시장 기사 요약
- 가격 수치 원본
- 일반 투자자 반응 해석

### 편집/검증 레이어
역할: 위 세 레이어의 결과를 합쳐 최종 브리핑을 만들고 검수한다.

권장 도구:
- OpenAI

담당 역할:
- 숫자와 해석의 일치 여부 검수
- 쉬운 한국어 변환
- 브리핑 구조화
- 제목/섹션/톤 통일

담당하지 않을 것:
- 새 데이터 수집
- 원본 수치 확정

## 5. 루트별 소유권과 중복 제거 규칙
### 원칙 1. 숫자는 정형 수치 레이어만 쓴다
예:
- 미국 10년물 금리
- 나스닥 등락률
- 비트코인 가격
- ETF 순유입

Perplexity나 X에서 같은 숫자가 보여도, 최종 채택은 정형 수치 레이어 기준으로 한다.

### 원칙 2. 시장 기사와 해석은 금융 리서치 레이어가 담당한다
예:
- 왜 반도체가 강했는지
- 연준 발언이 시장에 왜 부담이 됐는지
- ETF 자금 흐름이 심리에 어떤 영향을 줬는지

이 부분은 Perplexity 결과를 중심으로 가져가고, OpenAI는 이를 더 읽기 쉬운 문장으로만 바꾼다.

### 원칙 3. 공식 발표는 공식 시그널 레이어가 우선한다
예:
- 기업 발표
- SEC 공지
- ETF 발행사 업데이트
- 공식 X 포스트

기사 내용과 공식 발표가 충돌하면 공식 시그널이 우선이다.

### 충돌 해결 우선순위
1. 공식 시그널 레이어
2. 정형 수치 레이어
3. 금융 리서치 레이어
4. OpenAI 편집 해석

#### 미국 기술주 / AI

- GOOGL
- META
- AMD

#### 반도체 공급망

#### 비트코인 ETF

- Fidelity

### Conditional Fetch
가격 급등락, 큰 뉴스, 데이터 품질 저하 시에만 추가 확인한다.
- AAPL
- MSTR
- COIN
- MARA
- SMCI
- 기타 변동성 확대 종목

### 공식 채널 우선순위
각 엔티티는 아래 우선순위로 본다.
1. 공식 홈페이지 / newsroom / IR
2. 공식 발표 페이지
3. 공식 X 계정

### 실현성 검증 기준
이 기획은 `공식 X가 있을 것`을 전제로 하지 않는다.

실제 운영에서는 아래 조건을 모두 만족한 채널만 allowlist에 올린다.
- 공식 웹사이트 또는 공식 newsroom / IR 페이지가 실제로 존재한다.
- 공식 페이지 안에서 소셜 채널이 직접 연결되거나, 공식 고객지원 / 소셜 안내 페이지에서 공식 계정이 명시된다.
- 계정명이 티커나 브랜드명과 비슷하다는 이유만으로는 allowlist에 넣지 않는다.
- 검증일과 검증 출처 URL을 함께 기록한다.

즉, allowlist는 `추정 목록`이 아니라 `검증된 채널 레지스트리`여야 한다.

추가로 X 수집은 아래 제약을 전제로 설계한다.
- `allowed_x_handles`만 사용한다.
- `allowed_x_handles`는 한 번의 요청에 최대 10개까지만 넣는다.
- 날짜 범위는 `from_date`, `to_date`로 제한한다.
- 기본 조회 범위는 최근 24~72시간으로 둔다.
- 결과는 citations와 원문 링크를 함께 저장한다.

### 검증된 채널 레지스트리 운영 방식
운영 문서나 설정 파일에는 최소한 아래 필드를 둔다.
- entity_id
- entity_name
- ticker
- primary_domain
- newsroom_or_ir_url
- x_handle
- x_verified
- verification_source_url
- verification_method
- verified_at
- notes
- x_search_group
- x_search_priority

권장 규칙:
- `x_verified=true`인 경우에만 X 수집을 허용한다.
- `x_verified=false`면 공식 웹 / newsroom / IR만 사용한다.
- `verification_source_url`이 없으면 공식 계정으로 취급하지 않는다.
- 분기마다 또는 계정 변경 이슈가 있을 때 재검증한다.
- `x_search_group`은 한 번의 요청에 최대 10개 핸들만 포함하도록 설계한다.
- `x_search_priority`는 핸들 수가 10개를 넘을 때 우선 조회 대상을 정하는 기준으로 사용한다.

### 현재 기준 운영 정책
- `공식 X를 항상 쓴다`가 아니라 `검증된 엔티티만 쓴다`.
- 현재 바로 X를 붙여도 되는 그룹은 `AMD`, `Fidelity`, `BlackRock / iShares` 수준으로 보는 것이 안전하다.
- `Apple`, `TSMC`, `ASML`, `Bitwise`, `Grayscale`, `ARK / 21Shares`는 공식 웹 / IR / ETF 페이지가 메인이고, X는 추가 검증 전까지 기본 경로에서 제외한다.
- 따라서 초기 버전의 공식 시그널 레이어는 `웹 / newsroom / IR 중심`으로 설계하고, X는 검증된 일부 엔티티부터 제한적으로 붙이는 것이 맞다.

### X 수집 실행 규칙
1. 검색 방식
- `allowed_x_handles` 기반 제한 검색만 허용한다.
- 자유 키워드 검색은 공식 시그널 레이어에서 사용하지 않는다.

2. 날짜 범위
- 기본값: 최근 72시간
- 이벤트성 확인: 최근 24시간
- 주간 요약성 확인: 최근 7일 이내 별도 배치 가능

3. 저장 항목
- entity_id
- x_handle
- post_url
- posted_at
- text
- citations
- fetched_at
- query_window
- verification_source_url

4. 채택 기준
- 공식 시그널로 채택하려면 `x_verified=true`여야 한다.
- citations에 원문 링크가 있어야 한다.
- 동일 내용이 공식 웹 / newsroom에도 있으면 웹을 우선 표시하고 X는 보조 출처로 둔다.
- X만 있고 웹 공지가 없으면 `공식 계정 기준`이라는 설명과 함께 제한적으로 사용한다.

5. 그룹 구성 원칙
- 한 그룹은 최대 10핸들 이하
- 그룹은 제품 섹션 기준이 아니라 `실제 x_search 호출 단위` 기준으로 설계한다.

## 7. 트렌드 점수 모델
이 제품은 `미국 기술주 + 비트코인`이라는 코어 정체성을 유지하되, 그날 실제로 시장을 움직인 섹터나 테마에 더 많은 비중을 주는 방식이 적절하다.

핵심 원칙은 다음과 같다.
- 고정 코어는 유지한다.
- 그날 시장을 실제로 움직인 섹터는 추가로 확대 커버한다.
- 완전 자유형이 아니라 `후보군 안에서 가중치만 조정`한다.

### 고정 코어와 동적 오버레이
- 고정 코어 60~70%
  - 거시 환경
  - 미국 증시 흐름
  - AI / 빅테크
  - 비트코인 시장
- 동적 오버레이 30~40%
  - 그날 상대강도와 뉴스 밀도가 높은 섹터
  - 그날 공식 시그널이 강하게 나온 섹터
  - 그날 변동성이 커서 `주목할 변수`가 된 섹터

### 트렌드 점수 계산 항목
각 섹터는 매일 아래 항목으로 점수를 계산한다.

1. 가격 상대강도: 30점
- 시장 전체 대비 상대적으로 강했는지 또는 약했는지
- 예: 반도체 프록시가 나스닥보다 더 강한지

2. 거래량 / 변동성 확대: 20점
- 전일 대비 거래량이 늘었는지
- 변동성이 실제로 커졌는지
- 단순 등락보다 `시장 관심이 집중됐는지`를 본다

3. 신뢰 가능한 뉴스 밀도: 20점
- Perplexity 리서치 기준으로 상위 도메인 기사 수가 얼마나 되는지
- 같은 주제를 여러 출처가 다루는지

4. 공식 시그널 밀도: 15점
- 공식 발표, newsroom, 규제기관, 공식 X 포스트가 있었는지
- 기사 해석이 아니라 원문 시그널이 있는지

5. 시장 영향도: 10점
- S&P500, NASDAQ, 반도체, BTC 등 핵심 자산군 해석에 실제 영향을 주는지
- 단발성 종목 뉴스인지, 시장 전체 해석을 바꾸는 뉴스인지 구분한다

6. 지속성: 5점
- 하루짜리 잡음이 아니라 2~5일 이상 이어지는 흐름인지

### 점수 구간별 의미
- 70점 이상
  - 오늘의 리더 섹터
  - 뉴스 쿼리 확장
  - 공식 채널 추가 확인
  - 메일 상단 요약과 섹션 해석에 적극 반영
- 50~69점
  - 기본 커버리지 유지
  - 관련 뉴스 1~2건 반영
  - `주목할 변수` 수준으로 다룸
- 50점 미만
  - 기본 언급만 유지
  - 시장 전체 해석을 바꾸는 경우가 아니면 비중 확대하지 않음

### 운영 원칙
- 코어 섹터를 제거하지 않는다.
- 점수가 높다고 새로운 메인 제품으로 바꾸지 않는다.
- 동적 섹터는 `미국 증시 흐름`, `중요한 뉴스`, `시장 해석`에 우선 반영한다.
- 점수는 매일 바뀌지만, 후보군 자체는 자주 바꾸지 않는다.
- X 시그널 점수는 `x_verified=true`인 엔티티가 있는 섹터에서만 반영한다.
- 공식 X가 없는 섹터는 newsroom / IR / 공식 발표 밀도로 대체한다.

## 8. 섹터 후보군 / 대표 티커 표
아래 표는 트렌드 점수를 계산할 때 사용할 기본 후보군이다.

| 섹터/테마 | 대표 티커/엔티티 | 기본 역할 | 공식 채널 우선순위 |

| AI / 빅테크 | NVDA, MSFT, AMZN, GOOGL, META | 코어 섹터 | newsroom / IR -> 공식 X |

| 애플 생태계 / 소비 기술 | AAPL | 조건부 확장 | newsroom -> 공식 발표 -> 공식 X |
| 헬스케어 / 바이오 | LLY, NVO, UNH, XLV, IBB | 동적 확장 | IR / 발표 -> 공식 X |
| 금융 | JPM, GS, MS, XLF, KRE | 동적 확장 | IR / 규제 공지 -> 공식 X |
| 에너지 | XOM, CVX, XLE | 동적 확장 | IR / 발표 -> 공식 X |
| 산업재 / 인프라 | CAT, GE, ETN, PAVE | 동적 확장 | IR / 발표 -> 공식 X |
| 소비재 / 리테일 | WMT, COST, AMZN, XLY | 동적 확장 | newsroom / IR -> 공식 X |
| 방산 / 우주항공 | RTX, LMT, NOC, PLTR | 동적 확장 | IR / 발표 -> 공식 X |
| 소형주 / 경기민감 | IWM, regional banks, cyclicals | 시장 폭 확인용 | ETF/대표 종목 기준 |
| 비트코인 현물 | BTC | 코어 자산 | 가격 원본 + 규제/ETF 공식 소스 |

| 거시 / 규제 | Fed, Treasury, SEC | 코어 배경 | 공식 사이트 -> 공식 X |

### 후보군 운영 원칙
- AI / 빅테크, 반도체, 비트코인, 거시는 항상 본다.
- 헬스케어, 금융, 에너지, 산업재 등은 트렌드 점수에 따라 비중을 조절한다.
- 특정 종목 단발 이슈보다 `섹터 전체 방향성을 설명할 수 있는 대표 티커`를 우선 쓴다.
- 대표 티커는 뉴스 수집용 키워드와 공식 채널 매핑의 기준점이 된다.
- 실제 X allowlist는 이 표 전체가 아니라, 이 표 중 `x_verified=true`인 엔티티만 별도 그룹으로 관리한다.

## 9. 메일 반영 방식
트렌드 점수는 메일 전체 구조를 뒤흔드는 데 쓰지 않고, `같은 구조 안에서 비중을 동적으로 조절`하는 데 사용한다.

### 상단 요약 반영
- `핵심 요약` 3줄 중 1줄은 그날 리더 섹터를 반영한다.
- 예:
  - 반도체가 시장을 주도했다면 반도체 관련 판단을 상단 요약에 올린다.
  - 헬스케어가 강했다면 `오늘 시장은 헬스케어가 방어주 역할을 했다`는 식으로 반영한다.

### 주요 지표 반영
- `주요 지표` 4개 내외 중 1칸은 동적 섹터 관련 숫자를 넣을 수 있다.
- 예:
  - SOXX
  - XLV
  - XLF
  - XLE
  - IWM

단, 코어 지표를 밀어내지 않도록 한다.
- NASDAQ, BTC, 미 10년물, 달러/VIX 중 일부는 항상 유지한다.

### 섹션별 반영 규칙
#### 거시 환경
- 동적 섹터가 거시 변수와 직접 연결될 때만 반영한다.
- 예: 금리 상승으로 헬스케어보다 기술주가 더 크게 흔들리는 경우

#### 미국 증시 흐름
- 트렌드 점수가 가장 높은 섹터를 가장 먼저 반영한다.
- 지수 설명보다 `오늘 시장 리더십이 어디에 있었는지`를 강조한다.

#### AI / 빅테크 동향
- 코어 섹션으로 유지한다.
- 다만 시장 리더 섹터가 AI / 반도체가 아닐 경우, 길이를 약간 줄이고 핵심만 정리한다.

#### 비트코인 시장
- 코어 섹션으로 유지한다.
- 동적 섹터 변화와 직접 연결되는 경우에만 함께 언급한다.

#### 중요한 뉴스
- 3~5개 기사 중 1~2개는 그날 리더 섹터에서 가져온다.
- 나머지는 거시, 기술주, 비트코인, 규제처럼 코어 영향도가 큰 뉴스로 채운다.
- 공식 X 시그널은 기사 본문을 대체하지 않고, `공식 확인` 여부를 보강하는 용도로만 사용한다.

#### 시장 해석
- 트렌드 점수가 가장 높은 섹터를 `주목할 변수`에 직접 반영한다.
- 예:
  - 반도체 주도 장세가 다른 업종으로 퍼지는지
  - 헬스케어 강세가 방어 심리인지
  - 금융주 반등이 금리 안정 신호와 연결되는지

### 분량 조절 규칙
- 메일 총 길이는 유지한다.
- 동적 섹터가 늘어나면 코어 섹션을 없애는 것이 아니라, 중요도가 낮은 문장을 줄인다.
- 새 섹션을 매일 추가하지 않는다.
- 구조는 고정하고, 내용 비중만 조절한다.
- X 기반 공식 시그널은 메일 길이를 늘리는 별도 본문이 아니라, `핵심 이슈` 또는 `주목할 변수` 근거로 짧게 반영한다.

## 10. 분석 엔진 기획
단순 수집이 아니라 `판단 가능한 정보`로 바꾸는 분석 로직이 필요하다.

### 10-1. 거시 환경 분석
입력:
- 금리
- 국채
- 달러
- VIX
- Fed/Treasury/SEC 관련 뉴스와 공식 시그널

출력:
- 금리 부담이 커졌는지 완화됐는지
- 달러 강세가 기술주에 부담인지
- 위험선호 환경인지 보수적 환경인지
- 오늘 가장 중요한 거시 변수 1~2개

### 10-2. 미국 증시 흐름 분석
입력:
- 지수 등락
- 반도체 프록시
- 시장 뉴스
- 공식 기업 발표

출력:
- 지수 상승/하락보다 시장 폭이 넓은지 좁은지
- 반도체와 빅테크의 상대 강도
- 특정 테마 쏠림 여부
- 당일 리스크 포인트

### 10-3. AI / 빅테크 분석
입력:
- 대표 종목 가격
- Perplexity 리서치
- 기업 공식 소식

출력:
- AI 투자 기대가 강화됐는지 식었는지
- 실적/가이던스/CapEx 해석
- 종목별 차별화가 커지는지
- 오늘 주목할 기업 한두 곳

### 10-4. 비트코인 분석
입력:
- BTC 가격
- ETF 거래량
- ETF 보유량 / 순유입
- 공포탐욕지수
- 발행사 공식 정보
- 크립토 뉴스

출력:
- 가격보다 ETF 자금 흐름이 중요한지
- 심리가 과열/냉각 구간인지
- 가격과 거래량이 같이 가는지
- 오늘 가장 중요한 크립토 변수

### 10-5. 뉴스 우선순위 분석
뉴스는 많다고 좋은 것이 아니다.
다음 기준으로 3~5건만 뽑는다.
- 시장 영향도
- 공식 확인 여부
- 숫자 데이터와 연결 가능성
- 주제 다양성
- 중복 기사 여부
- citations 존재 여부

최종 뉴스는 아래 형태로 정리한다.
- 무엇이 있었는지
- 왜 중요한지
- 어느 섹션과 연결되는지
- 공식 시그널이 있는지
- citations가 있는지

## 11. 시각화 기획
이 프로젝트는 이메일이 메인이므로, 무거운 대시보드보다 `빠른 스캔과 해석 중심 시각화`가 맞다.

### 이메일에서의 시각화 원칙
- 이미지보다 HTML 기반 구조를 우선한다.
- 차트가 있더라도 텍스트 없이 의미가 전달되면 안 된다.
- 숫자와 해석이 항상 같이 붙어야 한다.
- 모바일에서 깨지지 않는 수준까지만 사용한다.
- citations가 있는 공식 시그널은 링크형 출처로만 짧게 노출하고, 이미지 스크린샷 의존은 피한다.

### 11-1. 상단 요약 영역
상단은 아래 세 블록으로 고정한다.
1. 날짜 + 제품명
2. 핵심 요약 3줄
3. 주요 지표 4개 내외

여기서 사용자는 첫 스크롤 안에 `오늘 시장이 어떤 날인지`를 파악해야 한다.

### 11-2. 섹션 카드 구조
각 섹션 카드는 아래 순서를 유지한다.
- 핵심 판단
- 주요 지표
- 배경과 해석
- 주목할 변수

이 순서가 중요한 이유는 다음과 같다.
- 먼저 결론을 읽고
- 숫자를 확인하고
- 맥락을 이해하고
- 오늘 무엇을 봐야 할지 알 수 있기 때문이다.

### 11-3. 수치 시각화 방식
이메일에서는 아래 수준이 적절하다.
- 상승은 빨간 텍스트
- 하락은 파란 텍스트
- 보합은 회색
- 숫자는 한 줄에 2개 이내
- 긴 표보다 인라인 정리

### 11-4. 향후 웹 버전 시각화
웹으로 확장할 때는 아래를 권장한다.
- 5일 mini trend line
- 1개월 비교 small multiples
- ETF 순유입 막대 + 누적선
- 기술주 relative strength matrix
- 뉴스 타임라인

단, 웹에서도 원칙은 같다.
- 차트 하나당 메시지는 하나
- 해석 없는 차트 금지
- annotation 우선

## 12. 2026 스타일 방향
### 방향 1. 마케팅 메일이 아니라 에디토리얼 브리핑
- 과한 배지, 장식, 감성 문구는 줄인다.
- `시장 메모`, `핵심 요약`, `주요 지표` 같은 언어를 쓴다.

### 방향 2. 높은 정보 밀도 + 낮은 피로감
- 정보는 많지만 한 화면에 과하게 몰아넣지 않는다.
- 카드 구분, 여백, 강약 조절로 읽기 피로를 줄인다.

### 방향 3. 차분한 전문성
- 지나치게 딱딱하지 않지만, 개인 블로그나 캐주얼 뉴스레터처럼 보이면 안 된다.
- 금융기관 morning note와 생활형 해설 사이의 중간 지점을 지향한다.

### 방향 4. annotation-first
- 차트가 있다면 선보다 설명이 먼저 보이게 한다.
- `오늘 왜 이런 숫자가 중요한지`가 먼저 전달돼야 한다.

### 방향 5. mobile-first
- 전체 제품의 기본 읽기 환경은 모바일이다.
- 첫 스크롤 안에 요약이 보여야 한다.
- 각 섹션은 길어도 20~30초 안에 스캔 가능해야 한다.

## 13. 전달 채널 설계
### 1차 채널: 이메일
현재 메인 채널이다.

장점:
- 매일 정시 발송 가능
- 사용자가 아침에 바로 확인하기 좋음
- 구조화된 요약에 적합

추천 구성:
- 헤더
- 핵심 요약
- 주요 지표
- 6개 섹션 카드
- 주요 참고 출처
- 짧은 면책 문구

### 2차 채널: 웹 아카이브
향후 확장 권장.

필요 이유:
- 이메일은 긴 차트와 히스토리를 담기 어렵다.
- 하루치 브리핑을 누적해 보기 어렵다.
- 사용자는 과거 대비를 보고 싶어한다.

추천 기능:
- 날짜별 브리핑 아카이브
- 섹션별 분리 보기
- 7일 / 30일 비교
- 차트와 주요 뉴스 타임라인

### 3차 채널: 요약 카드 / 소셜 발행
향후 선택 사항.

가능한 형식:
- 오늘의 3줄 요약
- 핵심 지표 카드
- 비트코인/기술주 요약 카드

단, 초기 버전에서는 메인 채널을 이메일에 집중하는 것이 맞다.

## 14. 운영 규칙
### 품질 게이트
브리핑 발송 전 최소 기준은 다음과 같다.
- 뉴스 4건 이상
- 도메인 3개 이상
- 공식 또는 상위 신뢰 출처 2건 이상
- 최신 기사 2건 이상
- 숫자 데이터 누락률 임계치 이하

### degraded 시 처리
- Perplexity 재질의
- legacy fallback
- 공식 시그널 추가 확인
- 그래도 부족하면 품질 알림 표시

### critical 시 처리
- 저장은 하되 메일 발송은 보류
- 이유를 로그와 결과 파일에 함께 남김

## 15. 단계별 실행 계획
### Phase 1. 데이터 소유권 분리
- 정형 수치 / 리서치 / 공식 시그널 / 편집 레이어를 명확히 나눈다.
- 각 레이어의 책임과 금지 영역을 고정한다.

### Phase 2. 대표 엔티티 watchlist 고정
- always-fetch / conditional-fetch 목록 확정
- 공식 채널 매핑 확정

### Phase 3. 분석 규칙 고도화
- 섹션별 핵심 판단 규칙 정의
- 뉴스 선별 규칙 고도화
- 충돌 우선순위 코드화

### Phase 4. 이메일 시각화 정교화
- 핵심 요약
- 주요 지표
- 섹션 카드
- 주목할 변수
- 참고 출처
- 디스클레이머 구조를 완성한다.

### Phase 5. 웹 아카이브 설계
- 아카이브 화면
- 날짜 비교
- 차트 모듈
- 출처 노출 구조 설계

## 16. 성공 지표
제품이 잘 설계됐는지는 아래 지표로 본다.
- 메일 오픈 후 첫 스크롤에서 핵심 파악 가능 여부
- 섹션당 평균 읽기 시간
- 뉴스/수치 품질 경고 빈도
- 사용자의 반복 열람률
- 대표 엔티티 공식 시그널 반영률
- 숫자와 해석 충돌 발생률

## 17. 지금 기준 최종 권장안
현재 단계에서 가장 현실적인 구조는 아래다.
- 정형 수치 레이어는 계속 강화한다.
- Perplexity는 금융 리서치 메인으로 사용한다.
- 공식 X는 대표 엔티티의 공식 계정만 allowlist 기반으로 사용한다.
- OpenAI는 최종 편집과 검증만 담당한다.
- 이메일은 에디토리얼 금융 브리핑 구조로 유지한다.
- 웹은 이후 확장 채널로 설계한다.
